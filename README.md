# Диагностика качества карточек Wildberries

Две связанные DL-задачи на общих данных (товары + отзывы, ключ связи `nmId`), категории: очки, чехлы на телефон, вешалки

1. **Табличная (MLP):** регрессия среднего рейтинга карточки по её атрибутам. Метрики: RMSE / MAE
2. **Текст (RNN → Transformer):** классификация оценки отзыва (1–5) + аспекты недовольства (брак, размер, доставка, качество, цена). Метрика: macro-F1

Связь: табличная модель находит карточки в зоне риска по рейтингу, текстовая объясняет причины из отзывов. Подробно — в [business_scheme.md](business_scheme.md)

## Данные

- **Каталог WB** — живой парсинг публичной выдачи по 3 категориям (4000 товаров на категорию, 12000 всего). `catalog.wb.ru` переехал за антибот (HTTP 498), его **не обходим**; берём публичнуюпоисковую выдачу `search.wb.ru` (только rate-limit, без антибота). Правила сбора: только публично видимые данные, некоммерческое учебное использование, активные челленджи (PoW, капча) не решаем, без прокси-ротации и фейк-аккаунтов; клиент `curl_cffi` представляется обычным Chrome (User-Agent и TLS-стек), один поток с троттлингом и дневным лимитом
- **Отзывы** — CC0-датасет `wb-feedbacks`, только по `nmId`, которые есть в спарсенном каталоге (стратификация по оценке 1–5, всего до ~50–100k отзывов)
- Связь датасетов — по ключу `nmId`. Поля — [docs/data_dictionary.md](docs/data_dictionary.md)
- Данные в гит не коммитятся (`.gitignore`), лежат локально в `data/`

### Датасет отзывов (wb-feedbacks)

Публичный CC0-датасет на HuggingFace — [nyuuzyou/wb-feedbacks](https://huggingface.co/datasets/nyuuzyou/wb-feedbacks), примерно 194 млн отзывов, около 10 ГБ. Скачивается в `data/raw/wb-feedbacks/`:

```bash
pip install huggingface_hub
hf download nyuuzyou/wb-feedbacks --repo-type dataset --local-dir data/raw/wb-feedbacks
```

После скачивания там должны лежать файлы `feedbacks-00.json.zst` ... `feedbacks-17.json.zst`

### Сбор данных (этап 2)

```bash
.venv/bin/python -m src.parse_catalog     # каталог 3 категорий -> data/interim/catalog.parquet
.venv/bin/python -m src.filter_reviews    # отзывы по nmId каталога -> data/interim/reviews.parquet
.venv/bin/python -m src.make_snapshot     # снапшот -> docs/snapshot.md / docs/snapshot.json
```

Парсер собирает страницы (пауза+джиттер, `Retry-After`, дневной лимит), кэширует страницы в `data/raw/catalog/<категория>/` и возобновляем (повторный запуск не ходит в сеть за уже скачанным). `filter_reviews` читает `.zst` стримингом, не грузя файл в память

### Предобработка и EDA (этап 3)

```bash
.venv/bin/python -m src.prepare_tabular   # чистка каталога + сплиты -> data/processed/tabular_*.parquet
.venv/bin/python -m src.prepare_text      # чистка отзывов, аспекты, стратифицированные сплиты -> data/processed/text_*.parquet
```

Кодирование и скейлинг — в `src/preprocess.py` (`build_tabular_preprocessor`, фит только на train), правила аспектов недовольства — в `src/aspects.py`. EDA с графиками и бизнес-инсайтами — [notebooks/eda.ipynb](notebooks/eda.ipynb)

### Табличная модель (этап 4)

Всё обучение — в [notebooks/tabular_mlp.ipynb](notebooks/tabular_mlp.ipynb): baseline без сети (среднее, Ridge, бустинг), 4 конфигурации MLP, сетка lr/weight decay, финальная оценка на test и важность признаков. Каждый запуск — отдельный run в DagsHub (эксперимент `wb-tabular-rating`): гиперпараметры, метрики по эпохам, кривые обучения и веса модели. Сравнение конфигураций и выводы — [docs/tabular_mlp.md](docs/tabular_mlp.md)

### Текстовая модель (этап 5)

Всё обучение — в [notebooks/text_dl.ipynb](notebooks/text_dl.ipynb): четыре значимо разные архитектуры на одних сплитах — TF-IDF с логрегом (baseline без сети), свёрточная сеть, двунаправленный LSTM и дообучение трансформера `cointegrated/rubert-tiny2` через Trainer HuggingFace. Метрики по эпохам, кривые обучения, early stopping, подбор гиперпараметров, macro-F1 и confusion matrix. Каждый запуск — отдельный run в DagsHub (эксперимент `wb-review-sentiment`): гиперпараметры, метрики каждой эпохи, веса модели и токенизатор. Аспекты недовольства (брак, размер, доставка, качество, цена) и темы негативных отзывов через LDA — бизнес-вывод «почему низкий рейтинг» по 3 категориям. Сравнение архитектур и выводы — [docs/text_dl.md](docs/text_dl.md)

### Сводный отчёт и воспроизводимость (этап 6)

Финальный свод обеих задач — в [main.ipynb](main.ipynb). Он ничего не обучает: тянет результаты обоих экспериментов прямо из логов DagsHub, сводит их в единый итог, воспроизводит метрику победителя каждой задачи **только из логов** и связывает модели по nmId — табличная ранжирует каталог по прогнозному рейтингу, текстовая по отзывам тех же карточек показывает главную жалобу. Прогоняется сверху вниз с чистого ядра без ошибок

Воспроизводимость: в DagsHub лежат не только модели, но и сами сплиты train/val/test — их выкладывает скрипт `src/log_datasets.py` отдельным раном `dataset` в каждом эксперименте. `main.ipynb` скачивает сплиты и веса прямо из логов в `data/from_logs/` и пересчитывает метрику, так что любой результат повторяется без локальных данных

```bash
.venv/bin/python -m src.log_datasets    # выложить сплиты train/val/test в DagsHub (один раз)
```

## Структура репозитория

```
data/
  raw/         сырые данные (парсинг WB, wb-feedbacks)
  interim/     промежуточные (catalog.parquet, reviews.parquet)
  processed/   чистые датасеты и сплиты train/val/test
  from_logs/   сплиты, скачанные из DagsHub при прогоне main.ipynb
main.ipynb     сводный отчёт обеих задач, прогон только из логов
notebooks/     EDA и обучение обеих задач (eda, tabular_mlp, text_dl)
src/           код (utils, парсинг, предобработка, обучение, выгрузка данных в логи)
models/        обученные модели
docs/          data dictionary, снапшот, правила сбора, инсайты, отчёты по этапам
configs/       config.yaml — пути, seed, параметры
Dockerfile     одинаковое окружение для всех (Python 3.12 + зависимости)
```

## Окружение

Python 3.12, зависимости с зафиксированными версиями в `requirements.txt`

## Как запустить

```bash
python -m venv .venv
.venv/bin/pip install -r requirements.txt
```

Логирование экспериментов — удалённое на **DagsHub (MLflow-совместимый)**. Доступы — в `.env` (не в гите): `MLFLOW_TRACKING_URI`, `MLFLOW_TRACKING_USERNAME`, `MLFLOW_TRACKING_PASSWORD`. В скриптах обучения перед стартом рана задаём тег `mlflow.source.name` относительным путём — иначе MLflow сам запишет туда абсолютный путь скрипта с именем пользователя, и он уедет в удалённый трекинг. В трекинге лежат и параметры с метриками, и веса моделей, и сами сплиты train/val/test, поэтому любой эксперимент воспроизводится только из логов — см. [main.ipynb](main.ipynb)

Единое окружение для всей команды — через Docker:

```bash
docker build -t gp5 .
```

## Результаты

| Задача              | Модель                     | Метрика    | Значение        |
| ------------------- | -------------------------- | ---------- | --------------- |
| Табличная (рейтинг) | baseline (среднее)         | RMSE / MAE | 0.1511 / 0.1058 |
| Табличная (рейтинг) | градиентный бустинг        | RMSE / MAE | 0.1418 / 0.0871 |
| Табличная (рейтинг) | MLP [64]                   | RMSE / MAE | 0.1448 / 0.0900 |
| Текст (оценка 1–5)  | baseline (TF-IDF+логрег)   | macro-F1   | 0.5159          |
| Текст (оценка 1–5)  | CNN-text                   | macro-F1   | 0.4685          |
| Текст (оценка 1–5)  | RNN (BiLSTM)               | macro-F1   | 0.4591          |
| Текст (оценка 1–5)  | Transformer (ruBERT-tiny2) | macro-F1   | 0.5011          |

Все метрики — на test, подробные сравнения — [docs/tabular_mlp.md](docs/tabular_mlp.md) и [docs/text_dl.md](docs/text_dl.md), сводный итог обеих задач — [main.ipynb](main.ipynb)

Общий вывод по обеим задачам: глубокое обучение корректно работает и уверенно бьёт константу, но на нашем объёме данных не обходит классику — в табличной задаче градиентный бустинг (0.1418) чуть точнее MLP (0.1448), в текстовой TF-IDF с логрегом (0.5159) держит первое место, а дообученный ruBERT-tiny2 (0.5011) занимает второе место и остаётся выбором на вырост. Бизнес-ценность — в связке: табличная модель по атрибутам находит карточки в зоне риска по рейтингу, текстовая по отзывам тех же nmId объясняет причину (брак в очках, посадка в чехлах, качество в вешалках). Это позволяет Wb создать инструмент, который можно будет продавать продавцам для повышения рейтинга, а так же уменьшие процента возвратов
